# 规范

# 1.命名规范

# 1-1. 文件夹的命名

  • 多个单词一般采用横线连接 (kebab-case)命名形式

# 1-2. 文件命名

  • 单文件组件文件名称 单文件组件的文件名应该要么始终是单词大写开头 (PascalCase)，要么始终是横线连接 (kebab-case)。
    // bad
mycomponent.vue
myComponent.vue

// good
my-component.vue
MyComponent.vue
  • 紧密耦合的组件名funs 和父组件紧密耦合的子组件应该以父组件名作为前缀命名。
    // bad
components/
|- TodoList.vue
|- TodoItem.vue
└─ TodoButton.vue

// good
components/
|- TodoList.vue
|- TodoListItem.vue
└─ TodoListItemButton.vue

# 1-3. 样式命名

  • 类名语义化，如内容区域的class类名定义为content，页脚区域的class类名定义为footer。
  • 全局样式使用camelCase命名区分
  • 私有class类名以全小写横线连接 (kebab-case)书写。

# 1-4. 常量

  • 全大写，使用下横线(_)拼接

# 1-5. 变量

  • 命名格式为camelCase。
  • 变量命名时，能做到见名知义。避免出现单个字母或语义不清的命名。
  • 函数名应该是动词或者短语，代表某种行为，描述它们在做什么。(ex：getUser(){...}, setUser(){...})。
  • import 组件引入以PascalCase命名。(ex: import SearchForm from "./SearchForm.vue")。
  • 单文件组件名name以PascalCase书写。(ex: name: "BaseInput" )(eslint)

# 1-6. prop

在声明 prop 的时候，其命名应该始终使用 camelCase，而在模板中应该始终使用 kebab-case。

// bad
export default {
  props: {
    'greeting-text': String,
  },
};

// good
export default {
  props: {
    greetingText: String,
  },
};

# 2.注释规范

当开发Vue前端应用程序时，可以使用以下注释规范：

# 2-1. 文件顶部注释

每个Vue文件的顶部应该有一个注释块，包含作者、日期、文件版本号、文件描述等信息。示例：

/**
* @file: MyComponent.vue
* @author: John
* @description: This file contains the MyComponent component.
* @date: 2023-05-30
* @version: 1.0.0
*/

# 2-2. 模块注释

对于每个Vue组件，应该有一个注释块，解释其功能和用途。比如：

/**
* @module: MyComponent
* @description: This component is responsible for displaying the user profile.
* @props:
*   avatarUrl {String}: The URL of the user's avatar.
*   username {String}: The user's username.
*   bio {String}: The user's bio.
* @events:
*   onFollow {Function}: Fired when the user is followed.
*/

# 2-3. 方法注释

对于每个方法和生命周期钩子函数，应该有注释，说明其功能、参数、返回值。比如：

/**
* Loads the user's data and sets up the component.
* @param {Object} data - The user's data.
* @returns {Promise} - A Promise that resolves when the data is loaded.
*/
async setup(data) {
  // ... some code here ...
}

# 2-4. 行内注释

在代码中使用注释，在关键代码行旁边提供上下文信息或解释。但应避免过多使用行内注释，以避免代码的可读性受到影响。比如：

// check if the user is authenticated
if (this.$store.getters.isAuthenticated) {
  // log the user out
  this.$store.dispatch('logout')
}

总体来说，注释应该清晰、简洁、有用，并且遵循一致的格式。这将有助于提高代码可读性和可维护性。

# 3.代码风格

vue风格指南 (opens new window)

# 3-1. 缩进 (eslint)

缩进使用2个空格

// 4个空格
function notifyUsers(users) {
    users.forEach(user => {
        const userRecord = database.lookup(user);
        if (userRecord.isVerified()) {
            notify(user);
        }
    });
}

// 2个空格
function notifyUsers(users) {
  users.forEach(user => {
    const userRecord = database.lookup(user);
    if (userRecord.isVerified()) {
      notify(user);
    }
  });
}

# 3-2. 元素属性上，使用双引号，不要使用单引号(eslint)

# 3-3. 元素属性名全小写，用中划线做分隔符(eslint)

// bad
<div :myName="myName"></div>

// good
<div :my-name="myName"></div>

# 3-4. 优先使用es6语法

例如：解构赋值、展开运算符、...

# 3-5. 比较运算符(eslint)

  1. 使用 === 和 !== 而非 == 和 !=
  2. 说明：使用 === 可以避免等于判断中隐式的类型转换，出错更少和定位错误更快。

# 3-6. 一个函数应该做一件事，避免在一个函数中执行多个操作。

// bad
function notifyUsers(users) {
  users.forEach(user => {
    const userRecord = database.lookup(user);
    if (userRecord.isVerified()) {
      notify(user);
    }
  });
}

//good
function notifyVerifiedUsers(users) {
  users.filter(isUserVerified).forEach(notify);
}

function isUserVerified(user) {
  const userRecord = database.lookup(user);
  return userRecord.isVerified();
}

# 3-7. 三元表达式

二选一条件下 三元表达式与if…else是可以互换的 根据条件二选一给变量赋值的情况下，可使用三元表达式，其他复杂的逻辑处理不推荐 如果有更好的实现，尽量不要使用三元表达式

// bad
let score = val ? val : 0

// good
let score = val || 0

# 3-8. 布尔转换

避免不必要的布尔转换， 在if条件等情况中会自动转为布尔值，无需!!，而赋值仍需!!来强制转为布尔值

// bad
const status = 1
if (!!status) {
}
const isDelete = status > 0 ? true : false

// good
const status = 1
if (status) {
}
const isDelete = !!status // or const isDelete = Boolean(status)

# 3-9. 自闭合组件

在单文件组件中没有内容的组件应该是自闭合的。

<!-- bad -->
<my-component></my-component>

<!-- good -->
<my-component />

# 3-10. 组件选项的顺序

组件选项应该有统一的顺序。

export default {
  name: '',
  mixins: [],
  components: {},
  props: {},
  data() {},
  computed: {},
  watch: {},
  created() {},
  mounted() {},
  destroyed() {},
  methods: {},
};

# 3-11. 单文件组件顶级标签的顺序

单文件组件应该总是让顶级标签的顺序保持一致，且标签之间留有空行。

<template>
  ...
</template>

<script>
  /* ... */
</script>

<style>
  /* ... */
</style>
